# Decisions (ADR log)

> **Last updated:** 2026-06-05 · **Source files:** project-wide

Architecture Decision Records — one entry per significant decision, with the
context that motivated it and the consequences we accept. Newest decisions are
appended at the bottom. Format: **Date · Status · Context · Decision · Consequences.**

---

## ADR-0001 — A `Fact` carries its `source_url` by construction

- **Date:** 2026-06-05
- **Status:** Accepted
- **Context:** The product's whole value is *grounded* outreach. If groundedness
  is checked only by a late verification pass, ungrounded facts can flow through
  research and drafting and slip past on an off day.
- **Decision:** Make [`Fact`](data-models.md) validate a non-empty `http(s)`
  `source_url` in its constructor, so an ungrounded fact is *unrepresentable*.
- **Consequences:** Groundedness is enforced at the type boundary, not as an
  afterthought. Every claim flowing through the pipeline already has a citable
  source, and "where did this come from?" always has an answer. Callers must
  supply a real URL when they create a fact — which is the point.

## ADR-0002 — External providers sit behind thin client interfaces

- **Date:** 2026-06-05
- **Status:** Accepted
- **Context:** LLM and search vendors change pricing, models, and availability
  often; we want to swap or A/B-test them without touching agent logic, and we
  want fast tests.
- **Decision:** Put LLM, search, and fetch behind small interfaces
  (`LLMClient`, `SearchClient`, `fetch_page`) that normalize to our own types
  (e.g. `SearchResult`). Concrete adapters (Gemini, Groq, Tavily) are selected by
  configuration; the pipeline never imports a vendor SDK directly. SDKs are
  imported lazily. See [Clients](components/clients.md).
- **Consequences:** Vendor swaps are a config change; tests mock at the interface
  seam with no network; the agent stays provider-agnostic. A thin mapping layer
  per provider is the maintenance cost.

## ADR-0003 — Hybrid: deterministic outer graph + agentic research loop

- **Date:** 2026-06-05
- **Status:** Accepted
- **Context:** Outbound communication must be auditable and reproducible, but
  research genuinely benefits from open-ended exploration.
- **Decision:** Make the outer pipeline a deterministic
  [LangGraph](glossary.md) state machine (`research → qualify → draft → verify →
  log`) with explicit gates, and confine agency to a bounded
  [ReAct](glossary.md)-style sub-loop *inside* the research node (capped by
  `RESEARCH_MAX_QUERIES`). See [Architecture](architecture.md) and
  [Pipeline](pipeline.md).
- **Consequences:** We get the adaptability of an agent where exploration helps
  and the predictability of a state machine everywhere else. The cost is two
  execution models to reason about instead of one.

## ADR-0004 — No auto-send, no LinkedIn scraping

- **Date:** 2026-06-05
- **Status:** Accepted
- **Context:** Outbound tooling is easy to turn into a spam cannon, and LinkedIn
  scraping violates platform terms and invites bans.
- **Decision:** pitch-pilot never sends automatically — qualified, verified
  drafts are *enqueued for human review* and a person approves before anything
  goes out. LinkedIn scraping is excluded entirely. See
  [Limitations](limitations.md).
- **Consequences:** Stays on the right side of platform terms and anti-spam
  norms; the human-review gate enforces a quality bar; the project leads with
  *trustworthy, grounded* output rather than raw volume. Throughput is
  intentionally lower than a fully-automated sender.

## ADR-0005 — Documentation = curated narrative + auto-generated reference

- **Date:** 2026-06-05
- **Status:** Accepted
- **Context:** Docs that are written by hand and updated by memory drift out of
  sync with the code. We want thorough docs that *stay* true as later phases land.
- **Decision:** Split documentation into (a) hand-written narrative pages for
  design and concepts and (b) an API reference **auto-generated from Google-style
  docstrings** via [MkDocs Material](glossary.md) + `mkdocstrings`. The reference
  is generated by `scripts/gen_ref_pages.py` (`mkdocs-gen-files` +
  `mkdocs-literate-nav`), so new modules appear automatically with no manual nav
  edits. A maintenance protocol in `CLAUDE.md` ties every code change to its docs,
  and `mkdocs build --strict` gates each phase.
- **Consequences:** The reference can never drift from the docstrings. Narrative
  docs carry a "Last updated · Source files" header so drift is visible.
  Contributors must keep docstrings rich (they are the reference) and follow the
  `CLAUDE.md` protocol. Two small extra dev dependencies are accepted to make the
  reference self-maintaining.
