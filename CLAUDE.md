# CLAUDE.md — project instructions for pitch-pilot

**pitch-pilot** is an autonomous SDR agent: it researches a company from its
domain, qualifies it against an ICP, drafts grounded outreach, verifies every
claim against a source, and queues the result for human review. The hero
guarantee is **groundedness** — no `Fact` exists without a `source_url`. It never
auto-sends, and LinkedIn scraping is out of scope.

## Project map

| Path | What it is |
| --- | --- |
| `src/pitch_pilot/models/` | Typed pydantic contracts (`Fact` is the keystone) |
| `src/pitch_pilot/clients/` | Swappable LLM / search / fetch behind interfaces |
| `src/pitch_pilot/config.py` | `Settings` + cached `get_settings()` (fail-loud) |
| `src/pitch_pilot/graph/` | `PipelineState` + `build_pipeline()` (LangGraph, P1) |
| `src/pitch_pilot/nodes/` | Pipeline node functions (P1+) |
| `src/pitch_pilot/storage/` | `Store` protocol + `JsonStore` + review queue |
| `src/pitch_pilot/cli.py` | `python -m pitch_pilot.cli smoke` |
| `tests/` | Unit tests — **no network** |
| `docs/` + `mkdocs.yml` | Documentation site (this file governs it) |
| `evals/` | Offline evaluation harness (P4) |

## Conventions

- **Python 3.11+, pydantic v2**, `src/` layout, installable via `pip install -e ".[dev]"`.
- **Google-style docstrings on every public symbol** — they are the API reference.
- **Lazy vendor imports** inside clients; **never hit the network in unit tests**.
- **Fail loud** on missing config; **never crash the pipeline** on a bad page.
- Windows/PowerShell is a first-class dev environment.

## Common commands (Windows / PowerShell)

```powershell
.\.venv\Scripts\Activate.ps1
pip install -e ".[dev]"          # install package + dev/docs tools
pytest                            # run the unit suite (no network)
python -m pitch_pilot.cli smoke   # P0 acceptance gate (needs .env keys)
mkdocs serve                      # preview docs at http://127.0.0.1:8000
mkdocs build --strict             # gate: must pass before a phase is "done"
```

# Documentation Maintenance Protocol

**Documentation is part of the change, not a follow-up.** Every code change in
this repo must carry its documentation update in the same session. The docs are
split into (a) hand-written narrative pages under `docs/` and (b) an API reference
**auto-generated from docstrings** via `mkdocstrings` (so the reference never
drifts — keep the docstrings rich). Follow these rules:

1. **Changed a module?** Update its docstrings **and** the matching
   `docs/components/*.md`, and bump that page's `Last updated` header.
2. **New env var or setting?** Update `docs/configuration.md` **and**
   `.env.example` (and the `Settings` model + its `Attributes:` docstring).
3. **New or changed pydantic model?** Update `docs/data-models.md`.
4. **New or changed node or edge?** Update `docs/pipeline.md` **and**
   `docs/components/nodes.md`.
5. **Made a design decision?** Add a dated **ADR** entry to `docs/decisions.md`.
6. **Completed a phase?** Update `docs/roadmap.md` status **and** add a
   `CHANGELOG.md` entry.
7. **Ran an eval?** Update the numbers table in `docs/evals.md` **with the date**.
8. **A phase is NOT "done" until `mkdocs build --strict` passes** — no broken
   links, no missing referenced pages, no warnings.
9. **`README.md` stays the hub:** short, and links into the docs site. Don't grow
   it into a second copy of the docs.

## Documentation conventions

- Every narrative doc starts with a header:
  `> **Last updated:** YYYY-MM-DD · **Source files:** <paths this doc describes>`
- Use **Google-style** docstrings (`Args:` / `Returns:` / `Raises:` /
  `Attributes:`). Do **not** use Sphinx roles like `` :class:`X` `` — write a
  plain code span `` `X` `` so it renders cleanly under mkdocstrings.
- Don't put literal triple-backtick fences inside a docstring's prose (they break
  Markdown rendering) — describe them in words instead.
- The API reference pages are generated by `scripts/gen_ref_pages.py`; new modules
  appear automatically — you do **not** edit `nav` for reference pages.
